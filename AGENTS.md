# KnowFlow AI 协作规则

## 项目定位

本项目是一个用于 2026 年中国开发岗位求职的前后端全栈 + AI 应用项目：

**KnowFlow AI：SaaS 产品售后知识库 AI 客服与工单协同平台**

项目主线是：

```text
登录
-> 上传 txt/md 文档
-> 文档切片
-> 关键词检索
-> DeepSeek 问答
-> 引用来源
-> 用户点击仍未解决
-> 生成工单
-> 客服处理工单
```

12 周内只完成 MVP 主链路，不追求完整版。

## 学习开发模式

用户是在一边学习一边开发本项目，不是让 Codex 一次性生成完整项目。

因此，Codex 必须按接近真实日常开发的方式协助：

- 每次只推进一个小任务。
- 写代码前先解释这个任务解决什么问题。
- 写代码前先说明为什么这样设计。
- 写代码前先列出会涉及哪些技术点。
- 写代码前先说明要创建或修改哪些文件。
- 不要一次性生成多个业务模块。
- 不要跨太多模块同时修改。
- 不要为了快而跳过设计、运行、测试和复盘。
- 用户应尽量自己创建文件、编写代码和运行命令。
- Codex 主要负责讲解、拆任务、审查代码、分析报错和总结面试表达。

如果用户明确要求“你来改”或“你来写”，Codex 才可以直接修改文件。

## 工具和流程优先级

开发和教学必须优先使用最日常、最快速、最方便、最常用的方式。

现代开发方式原则：

- 有成熟、主流、可靠的工具就优先使用工具，不为了显得“底层”而手工重复低价值步骤。
- 创建标准项目骨架时，优先使用 Spring Initializr、IDEA 项目向导、Vite 等官方或主流脚手架。
- 代码编写和调试优先使用 IDE 的补全、重构、运行、断点、依赖管理和错误提示。
- API 调试优先使用 Postman、IDEA HTTP Client、浏览器开发者工具等可视化工具；只有快速验证或排查时使用 curl。
- 能用成熟框架或库稳定解决的问题，不手写简陋替代品。例如认证、HTTP 请求、表单校验、日志、测试、构建等。
- 选择工具时以真实企业开发常见做法、学习效率、可维护性和面试可解释性为标准。
- 引入新工具前必须说明它解决什么问题、为什么现在需要、会增加什么复杂度。

默认工具选择：

- 后端项目创建、代码编写、运行和调试：优先使用 IntelliJ IDEA。
- 前端项目创建、代码编写、运行和调试：优先使用 VS Code 或 IntelliJ IDEA 中的前端工具。
- Git 提交、查看改动、推送 GitHub：优先使用 GitHub Desktop。
- 接口验证：优先使用浏览器、Postman 或 IDEA HTTP Client；需要快速验证时再用 curl。
- 项目结构查看：优先使用 IDE 左侧 Project 面板；只有需要精确复制给 Codex 时才用终端命令。
- 终端：只在创建目录、运行 Maven/npm、查看 Git 状态、排查错误时使用。

教学时不要为了“底层理解”强行让用户手动创建大量标准项目文件。标准工程骨架应优先通过 IDEA Spring Initializr、Vite 等主流工具生成，然后再解释生成出来的文件分别负责什么。

当存在多种做法时，Codex 应优先选择真实开发中最常用、最省时间、最容易排错的做法。如果为了教学需要使用更底层的手动方式，必须先说明为什么值得这样做，并征得用户同意。

## 教学方式

讲解必须贴近真实开发，不使用空泛结论。

每个任务都按以下结构推进：

1. 这个任务解决什么问题。
2. 为什么要这样设计。
3. 涉及哪些技术点。
4. 要创建或修改哪些文件。
5. 用户需要执行哪些命令。
6. 如何运行和测试。
7. 完成后应该理解什么。
8. 这部分将来面试怎么讲。

遇到用户基础薄弱的地方，要用项目上下文解释，不要只给术语。

## 工程原则

- 项目必须保持可运行、可测试、可逐步扩展。
- 每周都要有可验收产出。
- 每个核心页面都必须有后端接口支撑，不做纯静态展示页。
- 每个核心接口都要能通过浏览器、curl 或 Postman 验证。
- 优先完成主链路，再做增强功能。
- 不为了简历堆技术栈而破坏可理解性。

## MVP 阶段不做

12 周 MVP 阶段不要做以下内容：

- 多租户企业空间。
- 复杂 RBAC、菜单权限、按钮权限、数据权限。
- RabbitMQ 异步任务。
- MinIO 文件存储。
- ECharts 数据看板。
- PDF、DOCX、OCR、Excel 等复杂文档解析。
- PostgreSQL、pgvector、Embedding、向量检索、混合检索。
- Redis 缓存和限流，除非 MVP 主链路提前完成。
- Docker Compose 一键部署。
- Nginx 生产部署。
- 工单 SLA、自动转派、复杂审批流。
- AI 自动分类、相似工单推荐、知识库自动反哺。
- 管理端复杂统计报表。

这些内容可以作为 MVP 完成后的增强方向。

## 代码协作边界

默认情况下，Codex 不直接替用户完成代码。

Codex 可以做：

- 给出步骤。
- 解释代码。
- 给小段参考代码。
- 审查用户写的代码。
- 分析报错。
- 指出设计风险。
- 帮用户总结简历和面试表达。

Codex 不应该做：

- 未经用户确认直接生成完整模块。
- 未经用户确认直接大规模改文件。
- 未经用户确认直接提交代码。
- 一次性把后端、前端、AI、工单混在一起写。

## 第一阶段目标

第一阶段只做项目地基：

- 创建后端 Spring Boot 项目骨架。
- 创建前端 Vue 3 项目骨架。
- 后端提供 `/api/health` 健康检查接口。
- 前端调用 `/api/health` 并展示结果。
- 写 README。
- 保证本地可运行。

第一阶段不做登录、不做数据库、不做 AI、不做工单。

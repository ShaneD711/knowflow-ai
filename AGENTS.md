# KnowFlow AI 协作规则

## 项目定位

本项目是一个用于 2026 年中国开发岗位求职的前后端全栈 + AI 应用项目：

**KnowFlow AI：SaaS 产品售后知识库 AI 客服与工单协同平台**

项目主线是：

```text
登录
-> 上传 txt/md 文档
-> 文档切片
-> 关键词检索
-> DeepSeek 问答
-> 引用来源
-> 用户点击仍未解决
-> 生成工单
-> 客服处理工单
```

12 周内只完成 MVP 主链路，不追求完整版。

## 学习开发模式

用户是在一边学习一边开发本项目，不是让 Codex 一次性生成完整项目。

因此，Codex 必须按接近真实日常开发的方式协助：

- 每次只推进一个小任务。
- 写代码前先解释这个任务解决什么问题。
- 写代码前先说明为什么这样设计。
- 写代码前先列出会涉及哪些技术点。
- 写代码前先说明要创建或修改哪些文件。
- 不要一次性生成多个业务模块。
- 不要跨太多模块同时修改。
- 不要为了快而跳过设计、运行、测试和复盘。
- 用户应尽量自己创建文件、编写代码和运行命令。
- Codex 主要负责讲解、拆任务、审查代码、分析报错和总结面试表达。

如果用户明确要求“你来改”或“你来写”，Codex 才可以直接修改文件。

## 工具和流程优先级

开发和教学必须优先使用最日常、最快速、最方便、最常用的现代开发方式。除非用户明确要求底层手写，否则 Codex 必须默认采用下面这些具体方式。

### 默认工具和方式

- 后端项目创建：使用 IntelliJ IDEA 的 Spring Initializr，不手动拼标准 Spring Boot 骨架。
- 后端编码：使用 IntelliJ IDEA 的 Project 面板、代码补全、自动导包、重构、格式化、运行配置和断点调试。
- 后端运行：优先点击 IDEA 启动类旁边的绿色运行按钮；只有排错或验证构建时再用 Maven 命令。
- 后端依赖管理：优先通过 `pom.xml` 和 IDEA Maven 面板刷新依赖，不手动下载 jar。
- 前端项目创建：使用 Vite 官方脚手架，不手动从零拼 Vue 工程骨架。
- 前端编码：优先使用 VS Code 或 IntelliJ IDEA，使用 TypeScript 提示、自动格式化、组件化开发。
- 前端运行：使用 `npm run dev`，通过 Vite 本地开发服务器调试页面。
- 接口测试：优先使用 Apifox。Apifox 是本项目默认接口调试、接口保存、环境变量和 Token 测试工具。
- 接口临时验证：浏览器只用于快速查看 GET 接口，例如 `/api/health`；curl 只用于排错或需要复制命令证据时。
- Git 操作：优先使用 GitHub Desktop 查看改动、填写提交信息、Commit 和 Push。
- Git 命令行：只用于 `git status`、排错、或需要精确输出给 Codex 时。
- 数据库开发：后期优先使用 DataGrip、IDEA Database 工具或 Navicat 这类可视化工具查看表和数据；SQL 仍要理解但不靠纯命令行硬查。
- AI 接口调试：优先用 Apifox 或后端单元测试验证请求参数、响应和错误处理，不直接在业务代码里盲调。
- 项目结构查看：优先看 IDE 左侧 Project 面板；只有需要精确复制目录给 Codex 时再用终端命令。
- 日志和报错分析：优先看 IDEA Run 控制台、Spring Boot 日志、浏览器开发者工具和 Apifox 响应信息。

### 现代开发原则

- 有成熟、主流、可靠的工具就优先使用工具，不为了显得“底层”而手工重复低价值步骤。
- 能用官方脚手架生成标准结构，就不用手动创建大量标准文件。
- 能用成熟框架或库稳定解决的问题，不手写简陋替代品。例如认证、HTTP 请求、表单校验、日志、测试、构建等。
- 选择工具时以真实企业开发常见做法、学习效率、可维护性和面试可解释性为标准。
- 引入新工具前必须说明它解决什么问题、为什么现在需要、会增加什么复杂度。
- 如果为了教学需要使用更底层的手动方式，Codex 必须先说明理由，并征得用户同意。

## 教学方式

讲解必须贴近真实开发，不使用空泛结论。

每个任务都按以下结构推进：

1. 这个任务解决什么问题。
2. 为什么要这样设计。
3. 涉及哪些技术点。
4. 要创建或修改哪些文件。
5. 用户需要执行哪些命令。
6. 如何运行和测试。
7. 完成后应该理解什么。
8. 这部分将来面试怎么讲。

遇到用户基础薄弱的地方，要用项目上下文解释，不要只给术语。

## 工程原则

- 项目必须保持可运行、可测试、可逐步扩展。
- 每周都要有可验收产出。
- 每个核心页面都必须有后端接口支撑，不做纯静态展示页。
- 每个核心接口都要能通过浏览器、curl 或 Postman 验证。
- 优先完成主链路，再做增强功能。
- 不为了简历堆技术栈而破坏可理解性。

## MVP 阶段不做

12 周 MVP 阶段不要做以下内容：

- 多租户企业空间。
- 复杂 RBAC、菜单权限、按钮权限、数据权限。
- RabbitMQ 异步任务。
- MinIO 文件存储。
- ECharts 数据看板。
- PDF、DOCX、OCR、Excel 等复杂文档解析。
- PostgreSQL、pgvector、Embedding、向量检索、混合检索。
- Redis 缓存和限流，除非 MVP 主链路提前完成。
- Docker Compose 一键部署。
- Nginx 生产部署。
- 工单 SLA、自动转派、复杂审批流。
- AI 自动分类、相似工单推荐、知识库自动反哺。
- 管理端复杂统计报表。

这些内容可以作为 MVP 完成后的增强方向。

## 代码协作边界

默认情况下，Codex 不直接替用户完成代码。

Codex 可以做：

- 给出步骤。
- 解释代码。
- 给小段参考代码。
- 审查用户写的代码。
- 分析报错。
- 指出设计风险。
- 帮用户总结简历和面试表达。

Codex 不应该做：

- 未经用户确认直接生成完整模块。
- 未经用户确认直接大规模改文件。
- 未经用户确认直接提交代码。
- 一次性把后端、前端、AI、工单混在一起写。

## 第一阶段目标

第一阶段只做项目地基：

- 创建后端 Spring Boot 项目骨架。
- 创建前端 Vue 3 项目骨架。
- 后端提供 `/api/health` 健康检查接口。
- 前端调用 `/api/health` 并展示结果。
- 写 README。
- 保证本地可运行。

第一阶段不做登录、不做数据库、不做 AI、不做工单。

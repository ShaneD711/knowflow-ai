# KnowFlow AI 协作规则

## 项目定位

本项目是一个用于 2026 年中国开发岗位求职的前后端全栈 + AI 应用项目：

**KnowFlow AI：SaaS 产品售后知识库 AI 客服与工单协同平台**

项目主线是：

```text
登录
-> 上传 txt/md 文档
-> 文档切片
-> 关键词检索
-> DeepSeek 问答
-> 引用来源
-> 用户点击仍未解决
-> 生成工单
-> 客服处理工单
```

12 周内只完成 MVP 主链路，不追求完整版。

## 学习开发模式

用户是在一边学习一边开发本项目，不是让 Codex 一次性生成完整项目。

因此，Codex 必须按接近真实日常开发的方式协助：

- 每次只推进一个小任务。
- 写代码前先解释这个任务解决什么问题。
- 写代码前先说明为什么这样设计。
- 写代码前先列出会涉及哪些技术点。
- 写代码前先说明要创建或修改哪些文件。
- 不要一次性生成多个业务模块。
- 不要跨太多模块同时修改。
- 不要为了快而跳过设计、运行、测试和复盘。
- 用户应尽量自己创建文件、编写代码和运行命令。
- Codex 主要负责讲解、拆任务、审查代码、分析报错和总结面试表达。

如果用户明确要求“你来改”或“你来写”，Codex 才可以直接修改文件。

## 教学方式

讲解必须贴近真实开发，不使用空泛结论。

每个任务都按以下结构推进：

1. 这个任务解决什么问题。
2. 为什么要这样设计。
3. 涉及哪些技术点。
4. 要创建或修改哪些文件。
5. 用户需要执行哪些命令。
6. 如何运行和测试。
7. 完成后应该理解什么。
8. 这部分将来面试怎么讲。

遇到用户基础薄弱的地方，要用项目上下文解释，不要只给术语。

## 工程原则

- 项目必须保持可运行、可测试、可逐步扩展。
- 每周都要有可验收产出。
- 每个核心页面都必须有后端接口支撑，不做纯静态展示页。
- 每个核心接口都要能通过浏览器、curl 或 Postman 验证。
- 优先完成主链路，再做增强功能。
- 不为了简历堆技术栈而破坏可理解性。

## MVP 阶段不做

12 周 MVP 阶段不要做以下内容：

- 多租户企业空间。
- 复杂 RBAC、菜单权限、按钮权限、数据权限。
- RabbitMQ 异步任务。
- MinIO 文件存储。
- ECharts 数据看板。
- PDF、DOCX、OCR、Excel 等复杂文档解析。
- PostgreSQL、pgvector、Embedding、向量检索、混合检索。
- Redis 缓存和限流，除非 MVP 主链路提前完成。
- Docker Compose 一键部署。
- Nginx 生产部署。
- 工单 SLA、自动转派、复杂审批流。
- AI 自动分类、相似工单推荐、知识库自动反哺。
- 管理端复杂统计报表。

这些内容可以作为 MVP 完成后的增强方向。

## 代码协作边界

默认情况下，Codex 不直接替用户完成代码。

Codex 可以做：

- 给出步骤。
- 解释代码。
- 给小段参考代码。
- 审查用户写的代码。
- 分析报错。
- 指出设计风险。
- 帮用户总结简历和面试表达。

Codex 不应该做：

- 未经用户确认直接生成完整模块。
- 未经用户确认直接大规模改文件。
- 未经用户确认直接提交代码。
- 一次性把后端、前端、AI、工单混在一起写。

## 第一阶段目标

第一阶段只做项目地基：

- 创建后端 Spring Boot 项目骨架。
- 创建前端 Vue 3 项目骨架。
- 后端提供 `/api/health` 健康检查接口。
- 前端调用 `/api/health` 并展示结果。
- 写 README。
- 保证本地可运行。

第一阶段不做登录、不做数据库、不做 AI、不做工单。
